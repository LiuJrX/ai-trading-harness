# 短线交易 Harness Agent 系统设计

## 1. 系统定位

本系统的目标是做一个“通过对话驱动的短线交易工程系统”。它类似 Codex 的工作方式：用户通过聊天表达目标、补充经验、发起任务、确认计划；系统把对话转成结构化任务、策略上下文、风控约束、执行计划和审计记录。区别是 Codex 操作代码仓库，本系统操作交易研究、交易计划、模拟账户和未来可能接入的真实交易执行链路。

本系统不是先做 UI，也不是先做一个问答机器人。第一阶段只讨论系统能力：如何把历史复盘、游资笔记、博主观点、个人交易经验、后续文字策略输入和 agent 自己的复盘结论整理成可识别、可加载、可验证、可迭代的策略资产，并用 Harness 约束 agent 在盘前、盘中、盘后持续执行同一套交易工作流。

系统初期应定位为“交易决策辅助 + 模拟验证 + 复盘优化”，不直接自动实盘下单。所有交易建议都必须带有策略来源、适用条件、风险条件和验证状态。任何涉及真实交易的动作都必须由人最终确认。

核心设计原则：

- 经验不是简单文本，而是带条件、证据、适用场景和版本的策略资产。
- 策略不是一次性总结，而是可被复盘结果持续修订的版本化规则。
- `daily_review_files/` 里的图片只是历史经验冷启动输入，长期系统应主要接收文字策略、交易记录、盘后复盘和 agent 生成的策略更新提案。
- 加载不是简单 RAG，而是根据市场状态、任务场景、策略类型和个股上下文进行编译。
- Agent 不是自由发挥，而是在 Harness 规定的边界、流程和验证回路中工作。
- 聊天不是简单 UI，而是系统的任务入口、状态查询入口、策略维护入口和人工确认入口。
- 系统主轴是经验策略，不是纯量化；量化、行情、舆情、新闻公告和龙虎榜工具应作为观察器、验证器和证据源接入。
- 真实交易能力必须分级开放，先完成研究和模拟闭环，再接人工确认式执行，最后才考虑更高自动化。

## 2. 对话驱动的系统边界

用户希望的是“像 Codex 一样通过聊天完成真实炒股相关工作”。因此系统应把聊天设计成一套操作协议，而不是把聊天当成普通问答。

一条用户消息进入系统后，应经过以下流程：

1. `意图识别`：判断用户是在新增策略、询问市场、制定计划、请求复盘、修改规则、查询持仓，还是准备交易。
2. `任务建模`：把自然语言转成结构化任务，例如 `create_premarket_plan`、`review_trade`、`propose_strategy_update`、`prepare_order_intent`。
3. `上下文编译`：根据任务、市场状态、策略版本、交易记录和风控规则生成 `context_pack`。
4. `Agent 执行`：由对应 agent 完成计划、复盘、观察、策略维护或交易意图生成。
5. `风控裁决`：涉及交易动作时，必须由独立风控流程判断是否允许继续。
6. `人工确认`：真实交易相关动作必须进入确认流程，不能由模型一句话直接下单。
7. `状态落盘`：把任务、输入、加载规则、输出、确认记录和结果写入审计日志。

系统要支持的对话不是“问答”，而是下面几类可执行会话：

- `策略维护会话`：用户说“以后退潮期不要做二次反包”，系统生成策略更新请求。
- `盘前计划会话`：用户说“帮我做今天盘前计划”，系统生成市场状态、启用策略、禁用策略和观察清单。
- `盘中观察会话`：用户说“这个票现在能不能上”，系统只基于盘前计划、当前状态和风控规则判断是否满足条件。
- `交易准备会话`：用户说“如果符合条件，帮我准备买入计划”，系统生成交易意图和风险说明，但不直接绕过确认。
- `盘后复盘会话`：用户输入交易结果，系统归因是策略问题、环境问题还是执行问题。
- `策略优化会话`：系统根据多个复盘样本提出版本更新候选。

## 3. 真实交易能力分级

为了让系统未来可以接近真实炒股，但不在早期把风险集中到模型输出上，建议采用 5 个成熟度等级。

### 3.1 L0 研究与知识整理

系统只整理历史经验、术语、策略卡片、风控规则和失败模式。此阶段不产生买卖建议，只建立策略资产。

### 3.2 L1 盘前计划与盘后复盘

系统可以生成盘前计划、禁用策略、观察条件和盘后归因，但不处理盘中交易指令。此阶段验证策略表达是否稳定。

### 3.3 L2 模拟交易与纸面账户

系统根据盘前计划和盘中输入记录模拟触发、模拟买入、模拟卖出和模拟盈亏。此阶段验证策略是否能在连续样本中保持一致。

### 3.4 L3 人工确认式实盘辅助

系统可以生成真实交易意图，例如买入标的、价格条件、仓位上限、失效条件和退出计划，但必须由人确认后才能进入交易执行网关。系统不得直接越过确认。

### 3.5 L4 受限自动执行

只有当策略、风控、数据链路、模拟结果、审计和异常处理长期稳定后，才考虑小额度、白名单策略、严格仓位限制下的自动执行。L4 不是 MVP 目标。

### 3.6 L5 不建议作为近期目标

完全自动化实盘交易不应作为当前阶段目标。自动化投资工具存在模型假设错误、输出不适合个人情况、过度优化、系统故障、异常订单和信息安全等风险。系统设计应默认保留人的监督、确认和停机能力。

## 4. 总体架构

系统可以分为 7 层。

### 4.1 原始资料层

原始资料层保存未经修改的资料，包括历史冷启动图片、后续文字策略输入、交易记录、盘后总结、交易截图、行情快照和人工补充说明。`daily_review_files/` 下的图片是为了让系统读取历史复盘经验而准备的初始材料，不代表长期系统都以图片作为输入。

要求：

- 原始输入必须保留，不做覆盖式修改。
- 每个来源文件必须有唯一 `source_id`。
- 每次抽取都要记录抽取时间、工具版本、处理状态和人工校对状态。
- 原始资料只作为证据源，不直接作为 agent 决策上下文。
- 长期新增策略优先通过文字输入进入系统，例如“新增一条策略观察”“修订某个买点条件”“今天盘后发现一个失败模式”。

### 4.2 经验抽取层

经验抽取层负责把不同来源的经验输入转成 agent 可识别的结构化对象。历史阶段需要从图片中提取文字、段落、主题和交易经验；长期阶段则主要处理用户直接输入的文字策略、盘后复盘、交易记录和 agent 自己生成的复盘结论。这里不能只做全文入库，而要拆成可追溯、可校对、可版本化的经验对象。

主要产物：

- `source_document`：一张图片、一段文字、一份交易记录或一篇复盘对应的来源文档。
- `note_section`：按标题、日期、主题拆分的段落块。
- `atomic_insight`：最小经验单元，例如“龙头补涨空间通常低于龙头”。
- `case_reference`：具体案例，例如某日期、某题材、某个股、某种走势。
- `term_definition`：术语定义，例如“中位股”“情绪标”“筹码交换”“一字板开路”。
- `strategy_update_request`：用户或 agent 提出的策略新增、修订、废弃或降级请求。

### 4.3 策略资产层

策略资产层把零散经验整理成可执行的策略资产。它是系统的核心，不是向量库的替代品，而是 agent 真正加载和推理的对象。

主要资产：

- `principle`：长期原则，例如“短线投机的核心是筹码交换”。
- `market_regime_rule`：市场环境判断规则，例如主升、分歧、修复、退潮、混沌。
- `strategy_card`：策略卡片，例如龙头分歧转一致、补涨套利、首板试错。
- `trade_playbook`：交易剧本，描述盘前、盘中、盘后怎么执行。
- `risk_rule`：风控规则，例如退潮期禁用、跌停扩散减仓、连续亏损停手。
- `failure_pattern`：失败模式，例如高位退潮接力、缩量反包骗炮、中位股补跌。

### 4.4 市场状态层

市场状态层每天生成当前交易环境快照。短线交易不能脱离环境判断，所以 agent 在加载任何策略前，必须先判断市场状态。市场状态可以来自人工输入、行情工具、量化统计、舆情工具和 agent 复盘，但最终必须沉淀成结构化 `market_snapshot`，不能让外部工具的原始输出直接驱动交易建议。

市场状态至少包括：

- 指数环境：强、弱、震荡、共振下跌。
- 短线情绪：主升、分歧、修复、退潮、混沌。
- 赚钱效应：连板、趋势、题材、容量核心、低位首板。
- 亏钱效应：跌停数量、大面股、中位股补跌、高标断板。
- 题材结构：主线、支线、轮动、退潮、空窗。
- 龙头状态：高度、辨识度、带动性、封单质量、分歧承接。
- 量化观测：涨停数量、连板高度、炸板率、跌停数量、成交额变化、板块强度。
- 舆情观测：新闻催化、公告事件、政策方向、社媒热度、热榜变化。

### 4.5 外部工具与数据层

外部工具与数据层负责接入行情、量化、舆情、公告、龙虎榜和交易软件等工具。它的定位是“感知与验证”，不是策略主脑。系统不能因为某个量化指标或舆情热度单独触发交易，必须回到策略资产、市场状态和风控规则中统一裁决。

工具类型：

- `行情工具`：指数、个股、分时、K 线、成交额、涨跌幅、涨停跌停、板块行情。
- `短线情绪工具`：涨停梯队、连板高度、炸板率、跌停数量、大面股、中位股亏钱效应。
- `量化工具`：条件统计、历史样本回看、策略触发频率、胜率、盈亏比、回撤和模拟交易。
- `舆情工具`：新闻、公告、政策、社媒热度、同花顺热榜、开盘啦题材热度、博主观点。
- `资金与席位工具`：龙虎榜、游资席位、机构买卖、主力资金流向。
- `交易工具`：未来可能接入同花顺、券商接口或其他执行通道。

这些工具的输出必须先进入结构化观察对象，例如：

```yaml
external_signal:
  signal_id: sig_20260511_001
  source_type: sentiment_tool
  source_name: 同花顺热榜
  trade_date: 2026-05-11
  related_theme: 机器人
  observation: "机器人题材热度进入前排"
  interpretation: "题材关注度上升，但仍需盘面资金承接确认"
  confidence: medium
  usable_for:
    - market_snapshot
    - theme_observation
  not_allowed_for:
    - direct_order_execution
```

接入原则：

1. 经验策略决定“什么情况可以做”。
2. 市场数据判断“现在是否接近这种情况”。
3. 舆情数据解释“题材为什么可能发酵或证伪”。
4. 量化工具验证“这条经验在样本里是否站得住”。
5. 风控规则决定“即使机会成立，也能不能做、做多少”。
6. 复盘系统决定“这次结果是否应该改变策略”。

### 4.6 对话与任务编排层

对话与任务编排层负责把用户聊天转成系统任务。它不直接给交易结论，而是决定“该由哪个 agent、带着哪些上下文、在什么权限下工作”。

主要组件：

- `Conversation Router`：识别用户意图和任务类型。
- `Task Planner`：把对话转成结构化任务、输入、输出和验收标准。
- `Context Compiler`：根据任务编译策略、市场状态、交易记录、外部信号和风控规则。
- `State Manager`：记录当前会话、任务状态、确认状态和后续待办。
- `Approval Manager`：管理真实交易相关确认流程。

### 4.7 Agent 工作流层

系统不应只设计一个全能 agent，而应拆成多个职责清楚的 agent。

- `对话编排 Agent`：理解用户意图，把聊天转成结构化任务和可执行工作流。
- `资料整理 Agent`：从历史图片、文字笔记、交易记录和盘后复盘中抽取文本、段落、术语和原子经验。
- `策略归纳 Agent`：把原子经验合并成策略卡片、风控规则和失败模式。
- `策略维护 Agent`：处理用户文字输入和 agent 复盘提案，生成策略新增、修订、废弃或观察建议。
- `盘前计划 Agent`：根据市场状态选择今日启用策略、关注方向和禁入条件。
- `盘中观察 Agent`：跟踪计划中的触发条件、风险条件和市场状态变化。
- `盘后复盘 Agent`：对照策略规则、交易计划和实际走势做归因。
- `策略优化 Agent`：根据复盘证据提出策略版本更新建议。
- `外部信号 Agent`：把行情、量化、舆情、公告和龙虎榜信息转成结构化观察，不直接给交易结论。
- `量化验证 Agent`：对策略条件做历史样本回看、模拟统计和规则有效性检查。
- `风控 Agent`：独立检查仓位、频率、回撤、退潮期和纪律风险。
- `交易意图 Agent`：只生成交易意图、条件单草案和退出计划，不直接执行真实交易。
- `执行网关 Agent`：未来负责对接券商或交易软件接口，但只接受已确认、已通过风控的结构化指令。

### 4.8 交易执行与风控层

真实交易相关能力必须独立成层，不能混在聊天 agent 内部。

主要组件：

- `Order Intent`：交易意图，不是订单。包含标的、方向、触发条件、仓位上限、失效条件、退出计划。
- `Risk Gate`：风控裁决。检查市场状态、策略权限、仓位、亏损、频率、黑名单和异常条件。
- `Approval Gate`：人工确认。记录确认人、确认时间、确认内容和最终动作。
- `Execution Adapter`：未来接同花顺、券商接口或其他交易通道。
- `Kill Switch`：全局停机和禁用交易能力的开关。
- `Audit Log`：记录对话、策略版本、上下文包、风控结果、确认结果和执行结果。

交易链路必须遵循：

```text
用户对话 -> 交易意图 -> 风控裁决 -> 人工确认 -> 执行网关 -> 执行回报 -> 复盘记录
```

禁止链路：

```text
用户对话 -> 模型判断 -> 直接下单
```

### 4.9 Harness 控制层

Harness 控制层规定 agent 的边界、流程、状态、权限和验证方式。

它必须回答：

- 当前任务是什么？
- 本轮允许读取哪些资料？
- 本轮必须加载哪些规则？
- 本轮禁止做哪些动作？
- 结果如何验证？
- 什么情况必须停止并请求人工确认？
- 中间状态和结论写到哪里？
- 当前系统处于 L0-L5 哪个交易能力等级？
- 本轮是否涉及真实资金、真实订单或账户敏感信息？
- 本轮使用了哪些外部工具信号，是否已转换成结构化观察？
- 外部信号是事实、解释、猜测还是待验证假设？

## 5. 经验输入与抽取方案

系统的经验输入分为两类：历史冷启动输入和长期持续输入。

`daily_review_files/` 下的图片属于历史冷启动输入。它们是为了方便读取过去的复盘经验而存在，不应被一次性总结成一篇文章，而应经过“证据化抽取 -> 结构化归档 -> 策略化编译 -> 人工校对 -> 版本化发布”。

长期持续输入不应依赖图片。后续人的新增策略、盘后感悟、交易教训和规则修订，应优先通过文字形式告知 agent；agent 自己的盘后复盘和策略优化建议，也应以结构化文本进入策略维护流程。

### 5.1 输入来源分类

系统应支持以下来源：

- `historical_image_note`：历史图片复盘，用于冷启动策略资产。
- `human_strategy_note`：用户文字输入的新策略、新观察、新纪律或新禁忌。
- `daily_trade_review`：每日盘后复盘，包括计划、执行、盈亏、归因和教训。
- `trade_record`：交易记录，包括买入、卖出、仓位、时间、理由和结果。
- `market_snapshot`：市场状态快照，包括情绪、题材、涨停梯队、亏钱效应。
- `external_market_data`：行情和短线情绪数据，包括指数、个股、板块、涨停梯队、成交额、炸板率。
- `external_sentiment_signal`：舆情和事件信号，包括新闻、公告、政策、热榜、社媒讨论。
- `external_capital_signal`：资金和席位信号，包括龙虎榜、游资席位、机构买卖、资金流向。
- `quant_validation_result`：量化验证结果，包括条件回看、模拟交易、胜率、回撤、样本数量。
- `agent_strategy_proposal`：agent 基于复盘样本提出的策略更新建议。

不同来源可以使用同一套结构化资产模型，但进入流程不同。图片需要先 OCR 和视觉切分；文字输入可以直接进入语义抽取；交易记录和市场快照应先标准化字段；外部工具输出必须先转成结构化观察或验证结果；agent 提案必须进入审核队列，不能直接覆盖正式策略。

### 5.2 历史图片预处理

每张图片先建立元数据：

```yaml
source_id: daily_review_20240609
file_path: daily_review_files/20240609.png
date_range:
  start: 2024-06-09
  end: 2024-06-09
image_size:
  width: 1728
  height: 9924
status: pending_extraction
```

长图需要按视觉段落切分，而不是直接整图 OCR。切分依据包括标题、日期、编号、空行、项目符号、字体层级和主题词。切分后的每段都要保留图片坐标，方便后续回溯原图。

### 5.3 OCR 与视觉理解

抽取分两步：

1. OCR 抽出原文，尽量保持换行、标题、编号和项目符号。
2. 视觉模型复核结构，识别哪些内容是标题、结论、例子、个人经验、风险提示、交易规则。

输出不应只有纯文本，而应是带结构的文档：

```yaml
source_id: daily_review_20240609
sections:
  - section_id: sec_001
    title: 成长赛道
    raw_text: "龙头：行业龙头是一定要关注的..."
    position:
      page: 1
      y_start: 60
      y_end: 1020
    section_type: concept_note
```

### 5.4 文字策略输入抽取

长期系统中，用户可以直接用自然语言告诉 agent 新增或修改策略。Agent 不应把用户文字直接覆盖到正式策略，而应先转成 `strategy_update_request`，再抽取原子经验、影响范围和需要验证的条件。

示例输入：

```text
以后如果市场处在退潮期，就算龙头早盘超预期，也不要直接重仓上，最多观察或轻仓试错，除非板块中位股没有继续补跌。
```

应转成：

```yaml
update_request_id: sur_20260511_001
source_type: human_strategy_note
intent: revise_risk_rule
target_assets:
  - rr_no_high_position_in_recession
  - st_leader_first_divergence_repair
raw_text: "以后如果市场处在退潮期..."
proposed_change:
  summary: "退潮期限制龙头超预期重仓参与，要求中位股补跌风险消失后才允许提高参与度。"
  affected_fields:
    - disabled_regimes
    - risk_controls
    - trigger_conditions
validation_needed:
  - 回看近阶段退潮期龙头超预期案例
  - 检查中位股补跌是否能作为有效过滤条件
status: pending_review
```

这样设计的重点是：人可以很自然地告诉 agent 新策略，但系统内部必须走审核、验证和版本化流程。

### 5.5 Agent 自主复盘抽取

Agent 盘后复盘时可以提出策略优化，但只能形成提案，不能直接修改正式策略。提案必须带证据。

示例：

```yaml
proposal_id: asp_20260511_001
source_type: agent_strategy_proposal
proposal_type: tighten_entry_condition
target_strategy: st_leader_first_divergence_repair
reason: "最近 5 次龙头分歧转一致中，板块中位股继续补跌的 3 次均失败。"
evidence:
  postmarket_review_ids:
    - pmr_20260506
    - pmr_20260508
    - pmr_20260511
proposed_change:
  add_avoid_condition: "中位股继续补跌时，不参与龙头分歧修复的重仓确认。"
status: pending_human_review
```

策略优化 Agent 的职责是提出“候选修订”，不是替人做最终裁决。正式策略更新需要满足样本证据、冲突检查和人工确认。

### 5.6 原子经验抽取

每个段落再拆成 `atomic_insight`。原子经验是系统最小知识单位，必须一条只表达一个判断。

示例：

```yaml
insight_id: ai_20240609_001
source_id: daily_review_20240609
section_id: sec_001
type: market_mechanism
topic:
  - 龙头
  - 补涨
  - 行业周期
raw_quote: "补涨...但是涨幅的空间一定是低于龙头的。"
normalized_claim: "补涨股的上涨空间通常低于行业或题材龙头。"
applies_when:
  - 龙头已形成明确辨识度
  - 后排资金开始模仿龙头逻辑
does_not_apply_when:
  - 龙头尚未确认
  - 市场进入全面补涨或低位切换阶段
confidence: medium
needs_human_review: true
```

原子经验必须区分类型：

- `market_mechanism`：市场机制，例如资金如何从龙头扩散到补涨。
- `entry_rule`：买点规则，例如分歧转一致、打板确认。
- `exit_rule`：卖点规则，例如断板跌停、承接失败。
- `risk_rule`：风险规则，例如退潮期不参与高位博弈。
- `position_rule`：仓位规则，例如市场情绪弱时降低仓位。
- `regime_rule`：环境判断，例如主升、分歧、退潮。
- `failure_pattern`：失败模式，例如中位股补跌。
- `term_definition`：术语定义，例如情绪标、妖股、筹码交换。
- `strategy_revision`：策略修订观察，例如某个条件应加入、删除、放宽或收紧。

### 5.7 案例与经验分离

历史图片、文字复盘和 agent 自主复盘中经常会混合“结论”和“例子”。抽取时必须分离：

- 结论进入 `atomic_insight`。
- 具体日期、个股、题材、走势进入 `case_reference`。
- 二者通过 `evidence_links` 关联。

示例：

```yaml
case_id: case_20240609_001
source_id: daily_review_20240609
mentioned_entity:
  stock: null
  sector: 成长赛道
  concept: 龙头补涨
market_context: "行业龙头确认后，后排资金寻找补涨"
observed_pattern: "补涨空间低于龙头，且回调速度可能更快"
linked_insights:
  - ai_20240609_001
```

这样做的原因是：策略要从案例中归纳，但不能把单个案例直接当成规则。

### 5.8 术语归一

短线笔记里同一个概念可能有多种说法。系统必须建立术语表，让 agent 加载时使用统一语言。

示例：

```yaml
term_id: term_chouma_exchange
name: 筹码交换
aliases:
  - 换手
  - 接力换手
  - 承接
definition: "短线资金在分歧中完成持仓转移，新资金接住旧资金兑现筹码的过程。"
related_terms:
  - 分歧
  - 一致
  - 承接
  - 换手板
```

术语表非常重要。否则 agent 会把“分歧”“换手”“承接”“筹码交换”当成相近文本，而不是交易结构里的不同环节。

### 5.9 策略更新入口

长期系统必须提供一个明确的策略更新入口。所有新增和修订都进入同一队列：

```yaml
strategy_update_queue:
  - update_request_id: sur_20260511_001
    source_type: human_strategy_note
    priority: high
    status: pending_review
  - proposal_id: asp_20260511_001
    source_type: agent_strategy_proposal
    priority: medium
    status: pending_human_review
```

队列中的更新请求经过以下步骤：

1. 解析意图：新增策略、修订策略、增加风控、废弃规则、标记观察。
2. 定位影响：找到相关策略卡片、风控规则、失败模式和市场状态规则。
3. 冲突检查：检查是否与已有高优先级规则矛盾。
4. 证据绑定：绑定来源笔记、交易记录、复盘样本或人工理由。
5. 生成提案：形成可读的策略更新说明。
6. 审核发布：人工确认后写入正式策略版本。

## 6. 策略资产建模

经验抽取完成后，不能直接喂给交易 agent。必须编译成策略资产。

### 6.1 策略卡片

策略卡片描述“什么环境下，什么条件出现时，可以考虑什么动作”。

示例：

```yaml
strategy_id: st_leader_first_divergence_repair
version: 0.1.0
name: 龙头第一次分歧后的转一致
category: 龙头打法
status: draft
core_thesis: "龙头在主升或强分歧后的第一次有效分歧，如果次日出现超预期修复，可能形成高确定性的继续走强机会。"
enabled_regimes:
  - 主升
  - 强分歧后修复
disabled_regimes:
  - 退潮
  - 高位亏钱效应扩散
setup_conditions:
  - 龙头辨识度明确
  - 板块仍有梯队或助攻
  - 分歧后未出现彻底破位
trigger_conditions:
  - 竞价或开盘超预期
  - 分时承接强
  - 板块内助攻股修复
avoid_conditions:
  - 中位股连续补跌
  - 跌停数量扩散
  - 龙头封单或承接明显弱于预期
risk_controls:
  - 不在退潮期重仓参与
  - 触发承接失败时退出观察
evidence:
  insight_ids:
    - ai_20240609_010
    - ai_20240609_011
  update_request_ids:
    - sur_20260511_001
  proposal_ids:
    - asp_20260511_001
human_review: required
```

### 6.2 风控规则

风控规则必须独立于策略卡片，且优先级高于策略机会。

示例：

```yaml
risk_rule_id: rr_no_high_position_in_recession
name: 退潮期禁止高位重仓
priority: critical
condition:
  - 短线情绪 = 退潮
  - 高位股亏钱效应扩散
effect:
  - 禁用高位接力策略
  - 降低最大仓位
  - 盘前计划必须包含空仓选项
source_insights:
  - ai_20240609_025
```

风控规则必须由 `风控 Agent` 独立加载，不能被 `盘前计划 Agent` 因为看好机会而覆盖。

### 6.3 市场状态规则

短线策略必须先判断环境。市场状态规则用于把每日行情特征归类。

```yaml
regime_rule_id: mr_recession
name: 退潮期
signals:
  - 高标断板后负反馈加剧
  - 跌停或大面数量增加
  - 中位股补跌
  - 龙头失去带动性
  - 题材轮动加快且持续性差
effects:
  - 禁用多数高位接力策略
  - 提高空仓权重
  - 只允许低仓位观察或等待修复确认
```

### 6.4 失败模式

失败模式比成功案例更重要。失败模式用于盘前过滤和盘后归因。

```yaml
failure_pattern_id: fp_middle_stock_collapse
name: 中位股补跌
description: "龙头调整或市场退潮时，后排与中位股因缺乏稀缺性和辨识度，可能先于龙头快速补跌。"
warning_signals:
  - 中位股炸板率升高
  - 后排反包失败
  - 板块助攻消失
  - 资金只抱团少数高辨识度标的
avoid:
  - 高位中位股接力
  - 无辨识度补涨股追高
```

### 6.5 策略版本与审核状态

每个正式策略资产都必须有版本和状态。用户文字输入和 agent 自主复盘不会直接修改正式策略，而是生成候选版本。

```yaml
strategy_version:
  asset_id: st_leader_first_divergence_repair
  current_version: 0.1.0
  candidate_version: 0.2.0
  candidate_status: pending_human_review
  change_summary: "加入退潮期中位股补跌过滤条件。"
  change_source:
    - human_strategy_note
    - agent_strategy_proposal
  required_checks:
    - conflict_check
    - sample_review
    - risk_rule_review
```

状态建议：

- `draft`：从历史经验或文字输入初步生成，尚未用于正式决策。
- `active`：已审核，可被盘前和盘中 agent 加载。
- `watching`：可用于观察和复盘，但不能直接触发交易建议。
- `deprecated`：已废弃，只保留历史追踪。
- `pending_human_review`：等待人工确认。

## 7. 加载机制设计：不要简单 RAG

系统加载策略时，不应以“文本相似度”作为主路径，而应使用“场景路由 + 规则编译 + 证据补充”的方式。

### 7.1 常驻上下文

每个交易相关 agent 默认加载：

- 系统边界：不自动实盘下单，不承诺收益，不绕过人工确认。
- 交易纪律：先判断环境，再看策略；先风控，再机会。
- 风控底线：退潮期、连续亏损、仓位上限、敏感操作升级。
- 输出格式：必须说明策略来源、适用条件、风险条件和置信度。
- 当前交易能力等级：例如 L1 只允许计划和复盘，L3 才允许生成实盘交易意图。

常驻上下文不应太长，应该像宪法一样稳定。

### 7.2 市场状态路由

每天先由 `市场状态 Agent` 生成市场状态：

```yaml
trade_date: 2026-05-11
market_regime: 修复
confidence: medium
active_themes:
  - 机器人
  - AI应用
risk_flags:
  - 中位股亏钱效应未完全消失
  - 高标分歧加大
allowed_strategy_groups:
  - 龙头确认
  - 低位首板试错
disabled_strategy_groups:
  - 高位无确认追涨
```

后续 agent 只能加载与当前市场状态兼容的策略。若市场状态置信度低，系统默认降低策略激进度。

市场状态 Agent 可以使用外部工具，但必须区分事实和解释：

- 事实：涨停数量、连板高度、成交额、跌停数量、题材热榜排名。
- 解释：资金在高低切、题材在发酵、亏钱效应扩散。
- 假设：某个消息可能成为新主线、某个方向可能进入补涨阶段。

事实可以直接进入 `market_snapshot`；解释必须带依据；假设只能作为观察项，不能直接驱动交易建议。

### 7.3 对话任务路由

聊天入口必须先路由任务，再加载策略。

```yaml
conversation_turn:
  user_message: "今天这个龙头竞价超预期，能不能上？"
  detected_intent: intraday_trade_assessment
  required_context:
    - current_market_snapshot
    - premarket_plan
    - active_strategy_cards
    - risk_rules
    - position_state
  forbidden_actions:
    - direct_order_execution
  output_type: trade_intent_assessment
```

如果用户消息涉及真实交易，系统必须先判断当前能力等级是否允许。如果系统仍处于 L0-L2，只能输出研究或模拟建议；如果处于 L3，也只能生成待确认的交易意图。

### 7.4 任务场景路由

不同任务加载不同内容：

- 盘前计划：加载市场状态规则、外部市场观察、可用策略卡片、禁用策略、候选股过滤条件。
- 盘中观察：加载触发条件、撤退条件、分时确认规则、外部实时观察、风控规则。
- 盘后复盘：加载交易计划、实际走势、策略规则、失败模式、归因框架。
- 策略优化：加载一段时间内的复盘样本、命中/失败案例、策略版本历史。
- 策略维护：加载用户文字输入、agent 提案、相关策略版本、冲突规则和审核标准。
- 资料整理：加载抽取 schema、术语表和人工校对标准。
- 交易意图：加载盘前计划、当前市场状态、持仓状态、仓位规则、风控规则和人工确认要求。
- 量化验证：加载策略条件、历史样本、市场状态标签、交易记录和统计指标。
- 舆情观察：加载事件来源、题材关联、热度变化、盘面承接证据和风险提示。

这意味着同一条经验不会每次都被塞进上下文，而是在合适任务中以合适形态出现。

### 7.5 策略编译器

在每次 agent 执行前，由策略编译器生成 `context_pack`。它不是原始检索结果，而是一份经过筛选、排序、压缩和冲突处理的上下文包。

```yaml
context_pack_id: cp_20260511_premarket
task: premarket_plan
market_regime: 修复
loaded:
  principles:
    - 先判断市场环境，再判断个股机会
    - 风控优先于机会
  enabled_strategies:
    - st_leader_first_divergence_repair@0.1.0
    - st_low_position_first_board_trial@0.1.0
  disabled_strategies:
    - st_high_position_blind_relay@0.1.0
  risk_rules:
    - rr_no_high_position_in_recession
  failure_patterns:
    - fp_middle_stock_collapse
  external_signals:
    - sig_20260511_001
  quant_validation_results:
    - qvr_20260511_001
evidence_policy: "只展开与今日策略相关的证据，其他只保留引用 id"
```

策略编译器只加载 `active` 策略。`draft`、`watching` 和 `pending_human_review` 的策略可以出现在复盘或研究上下文中，但不能直接驱动盘中交易建议。

### 7.6 冲突处理

短线经验里一定会有冲突，例如“龙头要敢上”和“退潮期不要接高位”。系统必须有优先级。

默认优先级：

1. 风控规则高于策略机会。
2. 当前市场状态高于历史成功经验。
3. 明确失败模式高于模糊机会描述。
4. 人工确认的正式策略高于 agent 自主提案。
5. 盘面事实高于舆情解释。
6. 已验证策略高于未校对原子经验。
7. 足够样本的量化验证高于单日个案。
8. 近期同环境复盘证据高于跨环境泛化经验。

如果冲突无法解决，agent 必须输出冲突点并请求人工确认。

## 8. 每日运行流程

### 8.1 盘前

输入：

- 昨日市场数据
- 量化统计和短线情绪指标
- 舆情、公告、热榜和题材催化
- 最新复盘记录
- 策略资产库
- 当前关注池

输出：

- 今日市场状态判断
- 外部信号摘要和可信度
- 今日启用策略
- 今日禁用策略
- 候选方向和观察标的
- 触发条件
- 放弃条件
- 仓位约束

盘前输出必须明确“什么情况下不做”。

### 8.2 盘中

输入：

- 盘前计划
- 实时行情或人工输入的关键观察
- 实时外部信号或题材热度变化
- 触发条件和风控条件

输出：

- 当前是否符合计划
- 盘面与舆情是否共振
- 哪个条件被触发
- 哪个风险正在出现
- 是否需要继续观察、放弃或人工确认

盘中 agent 不应临时创造新策略。它只能根据盘前计划和预设规则判断偏离与触发。若用户要求真实交易，盘中 agent 只能生成交易意图，并交给风控和确认链路。

### 8.3 盘后

输入：

- 盘前计划
- 实际走势
- 交易记录
- 当日市场状态变化
- 当日量化统计与外部信号变化

输出：

- 计划是否有效
- 执行是否符合策略
- 盈亏来自策略优势、环境变化、执行偏差还是随机波动
- 舆情、公告或量化信号是否有实际盘面承接
- 哪些经验需要进入资产库
- 哪些策略需要观察或修订

盘后复盘必须区分三类错误：

- 策略错误：规则本身不适用或缺陷明显。
- 执行错误：规则有效，但没有按规则执行。
- 环境错误：市场状态判断错误，导致策略加载错误。

### 8.4 周期性策略优化

策略优化不应每天随意发生。建议按周或按样本量触发。

触发条件：

- 某策略累计出现足够样本。
- 同类失败模式重复出现。
- 某条风控规则多次避免亏损或多次过度限制机会。
- 市场风格发生明显变化。

输出必须是“策略更新建议”，不能自动覆盖正式策略。

### 8.5 人工文字策略更新

用户可以随时通过文字告诉 agent 新增经验或修改策略。系统处理流程必须是：

1. 接收原始文字，保存为 `human_strategy_note`。
2. 抽取原子经验和策略更新请求。
3. 判断影响哪些策略、风控规则或失败模式。
4. 生成候选修改和冲突说明。
5. 需要时回看历史复盘或模拟交易样本。
6. 人工确认后发布新版本。

示例：

```text
用户：以后龙头反包只看第一次，第二次反包除非板块强度特别超预期，否则不做。
```

Agent 不应直接把它变成永久规则，而应生成：

- 一条 `atomic_insight`
- 一个 `strategy_update_request`
- 一个影响范围列表
- 一个待验证问题：第二次反包在历史样本中的成功率是否显著降低
- 一个候选策略版本

### 8.6 交易意图与执行回报

当系统进入 L3 后，盘中聊天可以生成 `order_intent`，但不能直接生成裸订单。

```yaml
order_intent_id: oi_20260511_001
source_conversation_id: conv_20260511_intraday_003
strategy_id: st_leader_first_divergence_repair
symbol: "示例股票"
side: buy
intent_type: conditional_entry
conditions:
  - 市场状态仍为修复或主升
  - 龙头保持辨识度
  - 分时承接未破坏
  - 中位股未继续补跌
position_limit:
  max_position_pct: 10
invalid_when:
  - 跌停数量快速扩大
  - 板块助攻消失
  - 触发风控黑名单
risk_gate_status: pending
approval_status: required
execution_status: not_submitted
```

执行完成后，系统必须记录回报：

```yaml
execution_report:
  order_intent_id: oi_20260511_001
  submitted: true
  filled: true
  avg_price: null
  position_pct: null
  execution_channel: manual_or_adapter
  post_trade_review_required: true
```

真实交易链路中的每一次确认、拒绝、撤销和执行失败都必须进入盘后复盘。

## 10. 最小可行版本

第一阶段不需要直接接同花顺或开盘啦实时数据，也不需要开发 UI。更稳的 MVP 是一个命令行或脚本驱动的“聊天式系统内核”：

- 从 `daily_review_files/` 抽取 20 到 50 条高质量原子经验。
- 人工校对术语表和原子经验。
- 编译 3 到 5 张策略卡片，例如龙头打法、补涨、首板试错、退潮期风控。
- 建立文字策略更新入口，让用户可以用自然语言新增或修订策略，并生成候选版本。
- 建立对话任务路由，把用户消息转成结构化任务。
- 建立 `context_pack` 编译流程，让不同任务加载不同策略和风控上下文。
- 建立外部信号的结构化占位对象，先支持人工录入行情、量化观察和舆情观察，不急着接 API。
- 建立盘前计划模板和盘后复盘模板。
- 每天由人输入市场状态和候选股，agent 负责按策略卡片生成计划与复盘。
- 每天盘后由 agent 生成策略观察和优化提案，但不自动覆盖正式策略。
- 支持 L2 模拟交易记录，先验证触发、放弃、买入、卖出和复盘闭环。
- 连续运行一段时间后，再接行情数据和自动化观察。

MVP 验收标准：

- Agent 能说明每个建议来自哪条策略和哪条经验。
- Agent 能在退潮或风险环境下主动禁用策略。
- 盘后复盘能区分策略错误、执行错误和环境错误。
- 用户输入一条文字策略后，Agent 能生成候选更新、影响范围和待验证问题。
- Agent 自己提出的策略优化必须进入审核队列，而不是直接生效。
- 策略更新必须有样本证据，不允许凭单日情绪改规则。
- 用户通过聊天提出盘前、盘中、盘后任务时，系统能生成结构化任务和上下文包。
- 行情、量化或舆情输入能被记录为结构化观察，而不是直接变成交易建议。
- 涉及真实交易的对话必须被拦截为 `order_intent` 或研究建议，不能直接执行。

## 11. 后续演进

第二阶段可以完善文字策略维护流程，支持策略更新审核、版本对比、冲突解释和历史样本回看。

第三阶段可以完善对话式系统内核，支持多轮任务状态、任务恢复、审计日志和权限配置。

第四阶段可以接入市场数据，自动生成市场状态快照、涨停梯队、题材强度和亏钱效应指标。

第五阶段可以接入量化验证工具，对策略条件做历史样本回看、模拟交易统计、胜率、盈亏比和回撤分析。

第六阶段可以接入舆情、公告、热榜、龙虎榜和资金席位工具，用来辅助题材观察和催化验证。

第七阶段可以建立模拟交易账本，让策略在不实盘的情况下记录触发、放弃、买入、卖出和复盘结论。

第八阶段可以做半自动交易辅助：agent 给出计划、触发提醒、交易意图和风险提示，人做最终确认。

第九阶段才考虑更高自动化，但必须先满足长期稳定的策略验证、风控验证、回撤控制、异常处理和人工审计要求。

## 12. 关键结论

这个系统的核心不是“把笔记喂给模型”，也不是先做一个漂亮 UI，更不是把它做成纯量化交易系统，而是先做一个通过聊天驱动的经验策略交易内核。图片里的内容只负责提供历史经验种子；长期运行中，用户文字策略输入、交易记录、盘后复盘和 agent 自主复盘提案会持续进入系统。抽取层负责把这些输入拆成可追溯的原子判断；策略层负责把判断编译成可执行规则；外部工具层负责提供行情、量化、舆情、公告和资金观察；加载层负责在正确市场环境下调用正确策略；对话层负责把用户聊天转成结构化任务；交易执行层负责把真实资金动作隔离在风控、确认和审计之后。

真正有价值的 agent，不是每天给出更多观点，而是每天坚持同一套纪律：先判断环境，再选择策略；先看风险，再看机会；先记录证据，再优化规则。

真实交易能力的设计必须保守。系统可以越来越接近实盘，但每一步都要能回答：这次建议来自哪条策略，当前环境是否允许，风控是否通过，人是否确认，执行是否可追踪，盘后是否复盘。只有这些问题都有稳定答案，聊天式炒股 agent 才不是一个会说话的行情观点工具，而是一个可持续优化的交易工程系统。
