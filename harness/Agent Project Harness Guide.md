# Agent 项目 Harness 指导

## 1. 标题与定位

Harness 是 agent 的运行外壳，不只是 prompt。它定义 agent 可见的环境、可用的工具、允许的操作范围、执行流程、验证机制和失败恢复方式。开发任何一个 agent 项目之前，必须先设计 Harness，再设计任务细节。没有 Harness 的 agent，通常只能短时间回答问题；有 Harness 的 agent，才有机会持续完成工作。

本文件是通用工程准则，适用于任何 agent 项目的启动阶段。目标不是让模型“看起来更聪明”，而是让团队在项目开始前就把运行边界、验证回路、状态管理和恢复机制设计清楚，使 agent 的行为可执行、可验证、可恢复、可审计。

## 2. 为什么需要 Harness

Agent 的稳定性主要来自外部系统设计，而不是单次提示词优化。prompt 可以表达意图，但无法独立承担权限控制、状态管理、验证闭环和失败恢复。项目如果把可靠性建立在“模型这次应该能理解”上，最终会在长任务、多步骤任务和高影响任务中失稳。

长任务必须依赖状态、检查点和验证闭环，不能依赖上下文记忆。任何需要跨多个步骤推进的工作，都必须让 agent 在外部环境中持续记录计划、进度、决策、验证结果和阻塞点，否则任务会在中途偏航、重复劳动或在失败后无法恢复。

好的 Harness 会把 agent 从“会回答”变成“能持续完成工作”。它不是模型能力的装饰层，而是模型能力能够稳定落地的工程系统。项目真正要设计的，不只是“问 agent 什么”，而是“如何让 agent 在约束中推进、在验证中收敛、在失败后继续”。

## 3. 开工前必须明确的 6 类输入

### 3.1 目标与成功标准

项目启动前必须定义 agent 要完成什么，以及什么算完成。目标应具体到可交付结果，而不是停留在抽象意图。成功标准必须可验证，例如产出哪些文件、通过哪些检查、满足哪些功能或分析要求。没有成功标准的任务，不应交给 agent 持续执行。

### 3.2 范围边界

项目启动前必须明确 in-scope 和 out-of-scope。要说明 agent 可以做什么、不能做什么、在哪些情况下必须停下来请求人工确认。涉及产品方向、不可逆变更、需求解释冲突、外部依赖异常、权限不足或高风险操作时，必须定义清晰的升级条件，禁止 agent 自行越界决策。

### 3.3 工作区边界

项目启动前必须定义允许读写的目录和资源。需要明确哪些路径可修改、哪些只读、哪些禁止访问，是否允许生成中间文件，是否允许访问外部工作区或系统目录。工作区边界越明确，agent 误改无关内容的风险越低；边界不清楚时，默认收紧，而不是放宽。

### 3.4 工具与权限

项目启动前必须列清 agent 可用的命令、脚本、服务、网络能力和凭证来源。需要说明哪些工具是首选，哪些工具仅在特定场景下使用，哪些操作需要人工批准。对于网络访问、部署、数据库写操作、远程系统改动和敏感凭证读取，必须显式说明权限边界，禁止隐式假设。

### 3.5 验证方式

项目启动前必须定义每一步如何自测，最终如何验收。验证方式应覆盖静态检查、测试、脚本校验、人工检查点和最终交付确认。任何任务如果没有对应验证方法，就不应进入自动执行流程。agent 的默认行为应是“先验证，再宣布完成”，而不是“先完成，再看运气”。

### 3.6 交付形式

项目启动前必须定义结果输出到哪里、以什么格式交付、如何记录过程和结论。需要明确最终成果是代码、文档、报告、配置还是分析结论，存放位置是什么，过程状态记录在哪里，验收结论如何沉淀。交付形式不明确时，agent 很容易做出“看似完成、实际不可用”的输出。

## 4. Harness 的默认设计原则

### 4.1 先给环境，再给任务

Agent 在执行前必须先理解仓库、文档、入口、约束和常用命令。Harness 应优先提供环境说明，而不是直接抛出复杂目标。默认做法是先建立上下文，再开始执行；禁止让 agent 在未理解工作环境的情况下直接改动项目。

### 4.2 小步推进，步步验证

任务必须拆成可验证的小步，不能长链盲跑。每一步都应对应一个清晰目标、一个明确动作和一个可执行验证。默认做法是缩短推理链和执行链，把风险尽早暴露；禁止让 agent 在没有中间检查点的情况下连续推进多个高耦合动作。

### 4.3 外部状态优先

计划、进度、决策、待办、阻塞点和验收结果应写入文件或结构化状态，而不是只留在上下文中。Harness 应提供稳定的状态载体，让 agent 能回看历史、继续未完成工作并解释当前状态。默认做法是把关键状态外部化；禁止把长任务稳定性建立在上下文记忆上。

### 4.4 工具优先于推测

能通过命令、测试、搜索、日志和文件确认的事实，不应依赖模型猜测。Harness 应鼓励 agent 先观察、再判断、再行动。默认做法是使用工具建立事实基础；禁止在关键实现、依赖状态、文件位置或系统行为不确定时直接编造答案或实施变更。

### 4.5 默认安全

高风险操作必须显式确认，破坏性行为必须受限。Harness 应把删除、覆盖、回滚、远程写入、凭证使用和生产级变更纳入单独护栏。默认做法是先确认风险，再执行动作；禁止以“为了效率”为理由绕过安全边界。

### 4.6 可恢复

任何长任务都应能从检查点继续，而不是失败后重来。Harness 应要求任务分阶段产出中间结果，并在阶段结束后保存状态、记录验证结果和标注阻塞。默认做法是让失败变成可恢复事件；禁止把恢复成本转嫁给操作者重新解释整个任务。

### 4.7 结果可审计

关键动作、验证结果、失败原因和人工介入点必须能回看。Harness 应让团队在任务结束后回答三个问题：做了什么、为什么这样做、结果是否已验证。默认做法是保留足够的执行痕迹；禁止交付无法解释来源或无法追溯验证路径的结果。

## 5. 启动一个 Agent 项目时的 Harness 组成

### 5.1 项目说明

每个项目至少应提供一份项目说明，包含项目目标、关键目录、运行方式、常用命令、依赖前提和基本约束。它的作用是让 agent 在进入执行前先理解环境，而不是靠试错摸索项目结构。

### 5.2 Agent 工作说明

每个项目至少应定义 agent 的工作说明，明确允许做什么、禁止做什么、优先使用什么工作流、遇到什么情况必须停下。它是 agent 的行为边界，不应被埋在零散对话中。工作说明越清晰，执行越稳定。

### 5.3 任务状态载体

每个项目至少应准备一个状态载体，用于记录计划、进度、决策、问题、待办和验收结果。状态载体可以是文档、任务文件或结构化记录，但必须稳定、可读、可更新。没有状态载体的长任务，不应交给 agent 独立推进。

### 5.4 验证回路

每个项目至少应定义验证回路，覆盖 lint、测试、脚本检查、人工检查点和最终验收。Harness 必须明确哪些验证是默认必跑，哪些验证在特定变更后触发，哪些情况必须引入人工判断。没有验证回路的 agent 只能生产候选结果，不能生产可信结果。

### 5.5 升级机制

每个项目至少应定义升级机制，说明什么情况要停下并请求人工确认。升级条件通常包括目标冲突、需求歧义、权限不足、潜在破坏性操作、外部依赖异常、数据风险和验证失败。升级机制的作用不是降低自主性，而是防止 agent 在错误前提上高效推进。

### 5.6 失败恢复机制

每个项目至少应定义失败恢复机制，说明如何保存中间状态、如何重试、如何记录失败原因、如何避免重复劳动。恢复机制应覆盖命令失败、测试失败、上下文中断、依赖异常和部分完成状态。可恢复性是长任务 Harness 的基本要求，不是附加功能。

## 6. 开发任何 Agent 项目都要遵循的工作流

### 6.1 先理解环境

开始工作前，agent 必须先阅读仓库结构、入口文件、关键文档、运行方式和约束说明。若环境尚未理解，就不应进行高影响改动。默认顺序是先观察、再定位、再动手。

### 6.2 再确认任务

在开始执行前，agent 必须确认目标、边界、验收标准和交付形式。若任务描述与环境事实冲突，应优先澄清冲突，而不是带着模糊理解推进。

### 6.3 先做计划

Agent 必须把任务拆成若干可验证步骤，并明确每一步的产物和检查方式。计划不需要冗长，但必须足以支撑执行中的定位、恢复和沟通。

### 6.4 再执行

执行时每次只推进一个明确的小目标。对于会影响其他步骤的关键动作，应优先验证局部正确性，再继续后续步骤。默认做法是让执行链保持短而清晰。

### 6.5 每步验证

每完成一个关键步骤，agent 都必须运行对应检查并记录结果。验证失败时，应先分析失败来源，再决定修复、回退还是升级给人工。禁止在未验证状态下直接宣布完成。

### 6.6 持续记录

执行过程中，关键决策、障碍、偏差、临时结论和后续待办必须写入外部状态载体。记录不是为了留档而留档，而是为了让任务在上下文中断后仍能继续，并让他人能接手。

### 6.7 完成收口

任务结束时，agent 必须总结改动、验证结果、未解决风险和建议后续动作。交付不应只有结果，还应包含结果的可信度说明和剩余风险提示。

## 7. 必须内建的护栏

- 禁止擅自执行破坏性命令，包括删除、强制覆盖、硬回滚和不可逆批量修改。
- 禁止在不理解现有改动来源和意图的情况下覆盖用户工作。
- 禁止把未验证结果当成完成结果交付。
- 禁止在关键约束、权限边界或需求前提不清楚时继续推进高影响任务。
- 禁止把敏感信息写入日志、代码、文档、缓存或结果文件。
- 禁止在没有回退策略或升级路径时执行高影响变更。
- 禁止把“模型认为应该没问题”当作风险判断依据。

## 8. 长任务与多步任务的特殊要求

长任务必须分阶段推进，每个阶段都应有明确输出物，例如中间文档、状态记录、检查结果或局部成果。阶段边界越清晰，恢复成本越低。

每个阶段结束后都必须落盘状态，至少记录当前进度、已完成内容、验证结果、当前阻塞和下一步建议。上下文一旦中断，恢复应从状态文件开始，而不是从对话记忆开始。

遇到失败时，agent 必须记录卡点、相关上下文、已尝试动作、失败结果和下一步建议。失败记录的目标不是解释“为什么没做完”，而是让后续继续推进的人不必重复踩坑。

需要人工介入时，agent 必须清晰说明原因、当前状态和建议选项。升级信息应帮助人快速决策，而不是把模糊问题原样抛回去。

能并行的工作可以拆开，但写入边界必须清晰。多个执行单元如果会修改同一组文件或共享同一状态载体，必须先定义责任范围和合并顺序，避免相互覆盖。

## 9. 项目启动前检查清单

在任何 agent 项目开始前，团队至少应完成以下检查：

- 是否定义了目标和成功标准
- 是否定义了 in-scope / out-of-scope
- 是否定义了必须升级给人工确认的情况
- 是否列清了允许读写的目录与资源边界
- 是否列清了可用工具、权限限制和凭证来源
- 是否准备了项目入口说明和常用命令
- 是否准备了 agent 的工作说明与行为约束
- 是否定义了计划与进度记录位置
- 是否定义了每一步的验证方法
- 是否定义了默认必跑的检查项
- 是否定义了失败后的恢复方式
- 是否定义了最终交付格式和验收方式
- 是否定义了关键风险和对应护栏

若以上任一项仍不明确，默认不应直接进入长链执行阶段。

## 10. 结尾原则

一个好的 agent 项目，不是先问“模型能不能做”，而是先问“我们的 Harness 是否让它可以稳定地做”。当环境、边界、状态、验证和恢复都被设计清楚后，agent 才能真正成为可靠的执行系统，而不只是一个偶尔表现不错的回答器。
